# Ubuntu RAG Assistant

A fully local retrieval-augmented question-answering assistant for Ubuntu support questions. It retrieves vetted Ask Ubuntu Q&A with ChromaDB, generates answers with a local LLM via Ollama, exposes a Streamlit chat UI, and declines to answer when retrieval doesn't surface a strong enough match, instead of guessing.

## Overview

The assistant answers Ubuntu/Linux questions using a knowledge base of real, community-vetted Ask Ubuntu questions and their accepted answers. Everything runs locally: embeddings, the vector database, and the language model.

The defining feature is honesty about its own limits. A **relevance gate** checks how close the best retrieved match is to the question, and if even the closest result is too far away, the system returns "I don't have a good answer for that" instead of letting the model fabricate one. This makes a deliberate "I don't know" a first-class outcome, which, for a support assistant, is safer than a confident but wrong answer.

---

## How it works

The pipeline has a one-time build phase and a per-question query phase.

```
BUILD (once)
  QueryResults.csv ──► prepare_dataset.py ──► chroma_askubuntu/  (embeddings on disk)

QUERY (every question)
  question ─► retrieve top-k (cosine) ─► relevance gate ─► Ollama LLM ─► answer
                                              │
                                              └─ if best distance > 0.45 ─► "I don't have a good answer"
```

A key design point is that the **knowledge base and the test questions are two separate corpora**:

- **Knowledge base (trusted):** Ask Ubuntu questions with their _accepted_ answers, exported to `QueryResults.csv`. These are community-voted, so they serve as a trustworthy source of truth.
- **Test queries (noisy):** real user questions taken from the `sedthh/ubuntu_dialogue_qa` dataset (Ubuntu IRC support logs). These are messy and varied -- great as evaluation inputs, but not trustworthy enough to answer from, which is exactly why they are kept out of the knowledge base.

This mirrors how a real help desk works: questions come in, and the system retrieves from a curated knowledge base to answer them.

## Tech stack

- **Vector store:** ChromaDB 
- **Embeddings:** `sentence-transformers` — `all-MiniLM-L6-v2` (384-dim)
- **LLM (generation):** Ollama, running `qwen3:8b` locally
- **UI:** Streamlit (chat interface)
- **Evaluation data:** Hugging Face Hub (`sedthh/ubuntu_dialogue_qa`), pulled with `huggingface_hub` + `pandas`
- **Language:** Python 3.10+

## Project structure

```
ubuntu-rag-assistant/
├── prepare_dataset.py            # Builds the vector index from QueryResults.csv (run once)
├── rag_core.py                   # Shared logic: retrieval, prompt, relevance gate, model config
├── askubuntu_rag.py              # Terminal entry point -- quick single-question test
├── app.py                        # Streamlit chat UI
├── eval_harness.py               # Batch evaluation => CSV + HTML report
├── QueryResults.csv              # Ask Ubuntu Q&A export -- the knowledge base source data
├── chroma_askubuntu/             # Persisted Chroma vector index (generated by prepare_dataset.py)
├── eval_reports_for_ollama3b/    # Evaluation outputs for llama3.2:3b
├── eval_reports_for_qwen2_7/     # Evaluation outputs for qwen2.5:7b-instruct
├── eval_reports_for_qwen3/       # Evaluation outputs for qwen3:8b
├── requirements.txt              # Pinned dependencies
├── .venv/                        # Virtual environment (not committed)
```

The three `eval_reports_for_*` folders hold the per-model evaluation runs that back the model-selection write-up below. 

**The four scripts share one core.** `rag_core.py` holds the retrieval, prompt, relevance-gate, and model configuration; `askubuntu_rag.py`, `app.py`, and `eval_harness.py` all import from it, so the prompt or threshold is defined in a single place and every entry point stays in sync.

## Setup

**Prerequisites:** Python 3.10+, and [Ollama](https://ollama.com/) installed and running.

```bash
# 1. create and activate a virtual environment
python -m venv .venv
.venv\Scripts\activate          # Windows
# source .venv/bin/activate     # macOS / Linux

# 2. install Python dependencies
pip install -r requirements.txt

# 3. pull the local model
ollama pull qwen3:8b
```

The knowledge base source, `QueryResults.csv`, is an export of Ask Ubuntu questions and their accepted answers, produced with the [Stack Exchange Data Explorer](https://data.stackexchange.com/askubuntu/query/new) (a SQL query joining each question to its accepted answer, filtered by score).

## Usage

```bash
# Build the vector index (run once; re-run only if the data or settings change)
python prepare_dataset.py

# Quick single-question test in the terminal
python askubuntu_rag.py

# Launch the chat UI in the browser (http://localhost:8501)
streamlit run app.py

# Run the evaluation harness (writes a CSV + an HTML report)
python eval_harness.py
```

For the evaluation harness, set `GENERATE = True` in `eval_harness.py` to produce full answers for quality review; leave it `False` for a fast, retrieval-only pass when you only need the distance distribution to tune the gate threshold.

## Design decisions

A few choices that shaped the system, and why:

- **Cosine distance, not the default L2.** `all-MiniLM-L6-v2` produces length-normalised embeddings whose meaning lives in their _direction_, so cosine (angle-based) is the metric they were trained for. The Chroma collection is created with `configuration={"hnsw": {"space": "cosine"}}`.
- **Separating the trusted knowledge base from noisy test queries.** Vetted Ask Ubuntu accepted answers are the source of truth; the messy IRC-derived dataset is used only as evaluation input. Mixing the noisy data into the knowledge base would mean answering from unverified content.
- **A relevance gate for honest refusal.** Pure vector search always returns _something_, even for questions the corpus can't answer. The gate refuses when the closest match exceeds a cosine-distance cutoff (`0.45`), turning "I don't know" into a deliberate, safe outcome instead of a hallucination.
- **Evaluation-driven model choice.** The generation model was not picked by guesswork. A repeatable harness ran the same 30 questions through three models and the choice was made from the results — documented next.

## Model selection and evaluation

> The section below records how the generation model was chosen, based on a repeatable evaluation. The threshold-tuning and model comparisons were all run with `eval_harness.py`.

### Switching from `llama3.2:3b` to `qwen2.5:7b-instruct`

The generation model was first changed from Llama 3.2 (3B) to Qwen2.5 (7B instruction-tuned) after evaluating the system. This records the reasoning.

#### How the decision was made

I ran `eval_harness.py` over 30 real Ubuntu questions sampled from the `sedthh/ubuntu_dialogue_qa` dataset, with the relevance gate set to a cosine distance cutoff of `0.45`. For each question the harness recorded the best retrieval distance, whether the gate allowed an answer, and (with `GENERATE = True`) the generated answer.

#### What the evaluation showed

Retrieval and the relevance gate performed well. Low-distance hits were topically on point, and the six questions the gate refused (distance > 0.45) were genuinely weak matches or topics absent from the knowledge base, so refusing them was the correct behaviour.

The remaining problems were almost entirely on the **generation** side: the model received relevant context but produced flawed answers. Crucially, these failures occurred at _low_ retrieval distances — meaning retrieval succeeded — so no threshold change could catch them. They are model-quality issues, not retrieval issues:

- **Factual error (distance 0.407)** — asked what `sh` is, the model answered "the Bourne Again shell (Bash)". This is wrong: `sh` is the Bourne / POSIX shell, whereas _bash_ is the Bourne-Again shell.
- **Hallucination (0.432)** — asked how to avoid running as root permanently, the answer ended with advice to "throw away your hard drive and install ... Puppy Linux", which is nonsensical.
- **Context-summarising relapse (0.368)** — the "switch from Mint to Ubuntu" question returned "It seems like you've provided a collection of questions ... I'll provide a summary", followed by a structured digest of all the retrieved entries rather than a single answer. The cause is the _format_ of the retrieved context: each chunk is stored as `Title: ... Question: ... Accepted answer: ...`, so the "Reference snippets" block contained several entries that each begin with "Question:". The model read that stack of Q&A blocks as a set of questions to summarise — overriding the prompt's "do not summarise the snippets" instruction — and the longer, multi-topic chunks retrieved for this query made that misread especially likely. A generation failure driven by context format, not by retrieval.
- **Reasoning miss (0.349)** — for the `192.168.0.256` "unknown host" error, the model gave a plausible host-only-network explanation but missed the actual cause: `256` is an invalid IP octet (the valid range is 0–255).

#### Why a larger, instruction-tuned model

These are limitations of a 3B model: weaker factual precision, a higher tendency to hallucinate, and difficulty synthesising context made of several bundled Q&A snippets. The retrieval layer is already doing its job, so the lever for improvement is the generation step. Qwen2.5 7B Instruct is both larger and instruction-tuned, which targets these failures directly — more capacity for factual accuracy and reasoning, and stronger instruction-following to resist the "summarise the snippets" misread driven by the context format.

#### What stays the same

Only the generation model changes. Retrieval (cosine distance with `all-MiniLM-L6-v2`), chunking, the relevance gate, and the `0.45` cutoff are all unchanged. This keeps the upgrade a clean A/B test: the same 30 questions are re-run through `eval_harness.py` on `qwen2.5:7b-instruct` and compared against the Llama 3.2 baseline.

#### Trade-off

Qwen2.5 7B Instruct uses more memory and runs slower than the 3B model, especially on CPU. For a quality-focused support assistant this is an acceptable cost, since a wrong answer is more expensive than a slow one.

### Results after switching to Qwen2.5 7B Instruct

I re-ran `eval_harness.py` on the same 30 questions, with the gate left at `0.45` and retrieval and the prompt unchanged, so the only variable was the model. Checking the four failures:

- **Hallucination (was 0.432) — fixed.** The "throw away your hard drive and install Puppy Linux" answer is gone, replaced by a correct, specific `NOPASSWD:ALL` configuration via `visudo`, with a safety caveat.
- **Context-summarising relapse (was 0.368) — fixed.** The Mint→Ubuntu question now receives an actual pros/cons answer rather than "you've provided a collection of questions ... I'll provide a summary." The answer is verbose and somewhat generic, but it answers the question instead of digesting the snippets — the format-driven misread is resolved.
- **Reasoning miss (was 0.349) — fixed.** The `192.168.0.256` error is now correctly attributed to the IP exceeding the valid range.
- **Factual error (was 0.407) — improved but not fully fixed.** The `sh` answer is better phrased but still conflates `sh` with bash; the precise answer (that `sh` is the Bourne/POSIX shell, which on modern Ubuntu is `dash`) is still not given.

Two honest caveats. First, the swap shifts the error profile rather than eliminating errors: a new mistake appeared in the file-permissions answer, which recommended `chmod -R 644` across a directory tree — that strips the execute bit directories need and would break access. The 3B model's `find -type d` answer avoided that but was itself incomplete, giving only the directory command. So neither model fully nailed this question — the 3B was safer but incomplete, the 7B complete but buggy. They failed in different ways. Second, the relevance gate behaved identically — the same six questions were refused at the same distances — confirming the comparison was a clean A/B.

**Overall:** the larger, instruction-tuned model removed the hallucination and the summarising relapse and corrected the invalid-IP reasoning, a clear net improvement. It did not make the system perfect — one factual answer remained imprecise and one new error was introduced — which reflects that model capacity raises the quality floor without guaranteeing correctness.

### Next experiment: evaluating Qwen3 8B

The switch to Qwen2.5 7B Instruct resolved the most serious failures, but left two residual issues: the `sh`/bash conflation and the new file-permissions error. That residual gap on factual precision is the motive for testing one more model.

Qwen3 8B is a sensible candidate: it directly targets the remaining factual-precision weakness as the newer-generation successor to Qwen2.5 7B; it is a near-zero-cost experiment, running on the same hardware with a one-line change; and it turns the comparison into a methodical 3B → Qwen2.5 7B → Qwen3 8B progression rather than a single guess. The expectation is modest — this is a within-tier upgrade, so any gain is likely incremental and must be confirmed, not assumed.

To keep the comparison clean, only the model changes; the `0.45` cutoff, the prompt, retrieval, and the same 30 questions are held constant. Qwen3's "thinking" (chain-of-thought) mode is disabled, so it answers directly like the others — keeping the comparison fair and avoiding the long per-answer latency reasoning mode incurs on CPU.

### Final decision: three-model comparison and a prompt fix

Three generation models were evaluated on the same 30 questions, with the gate held at `0.45` and retrieval unchanged. The tracked cases:

|Case|Llama 3.2 3B|Qwen2.5 7B Instruct|Qwen3 8B|
|---|---|---|---|
|`sh` definition|wrong (calls `sh` "bash")|imprecise|correct|
|"sudo perpetually"|hallucinated (Puppy Linux)|correct and specific|no hallucination, but vague|
|Mint → Ubuntu|summarised the snippets|answered cleanly|answered, but verbose|
|invalid IP `…0.256`|missed the cause|correct|correct (clearest of the three)|
|recursive permissions|safe but incomplete|complete but buggy (`chmod -R 644`)|correct and complete|

**Qwen3 8B is the most factually accurate of the three.** It is the only model that got both the `sh` definition and the recursive-permissions command fully right, and it explained its correct answers most thoroughly.

It did, however, introduce a new class of problem — **output style** rather than correctness. Several answers were overlong, used emoji and markdown headings, ended with chatty sign-offs ("Let me know if you'd like help…"), and occasionally padded the response with information that was never asked for (one answer appended a whole "Why your system might be using more RAM" section). One answer (sudo) was also vaguer than Qwen2.5's, dropping the concrete `NOPASSWD` example.

So the trade-off shifted across the three models: the 3B was concise but error-prone; Qwen2.5 7B fixed the big errors but left one factual slip and a buggy command; Qwen3 8B fixed those too and is the most accurate, at the cost of verbosity and over-formatting.

**Decision: adopt Qwen3 8B.** For a support assistant, factual correctness is the property that matters most, and Qwen3 wins there. The style problems are addressable separately — they are a _prompting_ issue, not a model-capability issue. The generation prompt was therefore extended with explicit style constraints (answer concisely in plain prose; no emoji, headings, or lists; no sign-offs; answer only the question asked). This cleanly separates the two concerns: model choice fixed accuracy, and prompting controls output style.

Because the style fix is a prompt change, it was applied _after_ the three-model accuracy comparison, so that comparison itself stayed clean (only the model varied). The prompt constraint was then added and the evaluation re-run to confirm the style tightened without regressing accuracy.

Across all three models the relevance gate behaved identically — the same six questions were refused at the same distances — confirming that retrieval and the gate are independent of the generation model, and that each comparison was a clean A/B.

## Local vs AWS: is the managed version worth paying for?

The same system was built and run two ways, then evaluated on the same 30
questions. Held constant across both: the documents, the `all-MiniLM-L6-v2`
embeddings, the prompt, the relevance gate, and the `0.45` cutoff. Only two
things changed:

| | Local | AWS |
|---|---|---|
| Vector store | ChromaDB (on disk) | Amazon S3 Vectors |
| Generation | Qwen3 8B via Ollama | Claude Haiku 4.5 via Amazon Bedrock |
| Hardware | Intel Core Ultra 7 155U, 16 GB RAM, CPU only (no discrete GPU) | Amazon's hosted infrastructure, over the network |

Because only the store and the model differ, each axis below isolates one real
difference.

### 1. Retrieval — effectively identical

Same vectors in, so both stores returned the same neighbours at the same cosine
distances, often matching to three decimals. A handful of questions differed by
0.001–0.002, and one (the AirPort/Wine question) surfaced a different top
document — because ChromaDB and S3 Vectors use different approximate
nearest-neighbour implementations that can diverge at the margin when several
documents are similarly close. Crucially, no gate decision changed: the same
6 questions were refused and the same 24 answered on both backends. The store
swap changed retrieval only cosmetically.

### 2. Answer quality — Bedrock was best

Across the four models tried in this project (Llama 3.2 3B, Qwen2.5 7B, and
Qwen3 8B locally, and Claude Haiku 4.5 on Bedrock), Haiku 4.5 produced the
strongest answers. It got every previously-tracked failure right — the `sh`
definition, the "sudo perpetually" question, the invalid-IP reasoning, and the
recursive-permissions command — with no hallucination and clean, plain-prose
style (the style constraints in the prompt held). On one question where
retrieval was only tangential, it even flagged that it lacked a strong match
rather than overclaiming — a second layer of honesty on top of the gate.

### 3. Latency — AWS was ~33× faster on this hardware

| | Local (Qwen3 8B, CPU) | AWS (Haiku 4.5, Bedrock) |
|---|---|---|
| Median (answered) | ~109 s | ~3.3 s |
| Fastest | 21 s | 2.4 s |
| Slowest | 609 s (>10 min) | 4.4 s |

On a CPU-only laptop, local generation ranged from slow to unusable — a
ten-minute answer is not a support tool anyone would wait for. Bedrock returned
in about 3 seconds including the network round-trip.

This gap is **hardware-bound, not software-bound**: it reflects a consumer
laptop CPU versus datacenter accelerators. On a machine with a capable GPU,
local latency would be far lower — so the honest claim is "on CPU-only consumer
hardware, local generation is ~30–40× slower," not "local inference is
inherently slow." (Refusals were sub-second on both backends, since they skip
generation entirely.)

### 4. Cost

Claude Haiku 4.5 on Bedrock is **$1 per million input tokens and $5 per million
output tokens** (on-demand, us-east-1). At this project's scale:

- The full evaluation (24 generated answers, short prompts) cost **under $0.10**.
- A single answered query is roughly **$0.003**.
- At ~100 questions/day, generation would run on the order of **~$9/month**.
- S3 Vectors is pay-per-use with **no idle floor**, so storing 33K vectors and
  querying them costs cents — notably avoiding the ~$350/month minimum of a
  provisioned vector service such as OpenSearch Serverless.
- Embeddings run locally, so they add nothing.

Local generation has **no marginal cost**, but it isn't free: it requires the
hardware and, on CPU, minutes of compute per answer.

### Verdict

For this use case — a responsive support assistant on consumer hardware — **the
managed AWS version is clearly worth paying for.** It answered about 33× faster,
gave the best answers of any model tried, and cost pennies to run at this scale
with no idle infrastructure cost. The local version's advantages are that it is
free at the margin, fully offline, and keeps all data on the machine — which
matters for privacy-sensitive or air-gapped settings, or if a capable local GPU
would close the latency gap.

The deciding factor is context, not a universal winner:

- **Responsiveness, best quality, low ops, small scale → AWS.** Pennies buys
  datacenter-speed inference and a stronger model, with no hardware to own.
- **Zero marginal cost, offline, data-stays-local, or a GPU already on hand →
  local.**

For the goal that started this project — a support assistant a person would
actually wait for — the answer is yes: on a CPU laptop, the managed version is
worth its very small cost.
